# Skill 蓝图

## 名称建议

推荐名称：`premium-web-dna-workflow`

备选名称：

- `reference-driven-web-design`
- `premium-site-design-audit`
- `visual-dna-site-builder`

推荐使用第一个名称，因为它强调“提取设计 DNA + 工作流”，不会让 Skill 误解为复制参考页面。

## Skill 触发场景

适合触发：

- 用户提供参考视频、截图、网站，希望提炼高级网页设计方法。
- 用户要求把一个现有页面升级成更高级的工具型、产品型或品牌型网页。
- 用户明确说“不要照抄参考，但要学习它的方法/质感/工作流”。
- 用户需要先做设计审计、设计 DNA、可复用流程或实现计划。
- 用户要基于参考素材生成背景图/视频提示词，并用真实 HTML 叠加。
- 用户要求逐个 Section 实现并做桌面/移动/性能/动效验收。

## Skill 不应触发的场景

不适合触发：

- 只修一个后端 bug、API bug 或解析逻辑问题。
- 只做很小的 CSS 调整，例如改一个按钮颜色。
- 用户要生成一张独立图片，但不涉及网站设计。
- 用户要写文案、法务条款、README，而不涉及视觉或页面结构。
- 用户要求照搬某个网站完整布局、品牌或版权素材。
- 用户明确要求不要做设计分析，只要执行已确定的小改动。

## 推荐拆分

建议先做一个 Skill，不拆分多个。

原因：

- 参考素材分析、设计 DNA、背景媒体规范、HTML 叠加、Section 实现和验收是连续链路，强行拆开会丢上下文。
- 一个 Skill 可以内置三种模式：审计模式、设计模式、实现模式。
- 当前最容易出错的是“把参考图照抄”或“只做表面高级感”，一个 Skill 统一约束更安全。

未来如果内容过长，可拆成：

- `visual-dna-audit`：只做参考和现站审计。
- `premium-section-implementation`：只做逐段实现。
- `visual-qa-checklist`：只做截图、性能和响应式验收。

## Skill 执行步骤

### 1. 模式判断

先判断用户当前要的是：

- 审计：只读代码和参考资料，输出文档。
- 设计：输出方案、设计 DNA、提示词、实现计划。
- 实现：修改页面并验证。
- 验收：检查已有结果并列问题。

如果用户明确说“不要改代码”，Skill 必须进入审计或设计模式。

### 2. 保护现场

- 检查 Git 状态。
- 如果用户要求备份，先创建提交。
- 明确允许修改的目录。
- 如果视频或素材很大，提醒是否需要纳入 Git；如果用户已明确要求，则按项目策略处理。

### 3. 项目结构审计

- 读取项目目录。
- 识别技术栈。
- 读取页面入口、核心组件、样式系统、数据流、测试。
- 总结当前信息架构和核心用户路径。

### 4. 参考素材分析

- 检查 `ffmpeg` / `ffprobe`。
- 从视频中抽取关键帧。
- 保存关键帧、索引和联系表。
- 选择代表帧进行观察。
- 不逐帧 OCR，不凭文件名猜测。

### 5. 设计 DNA 提取

从关键帧和当前项目中提取：

- 色彩系统
- 字体系统
- 间距系统
- 圆角和边框
- 光影和背景
- 布局和视觉层级
- 动效原则
- 需要避免的问题

必须区分：

- 可复用于其他网站的通用方法。
- 仅适用于当前项目的专属规则。

### 6. 背景媒体规范

输出图片/视频提示词模板，并明确：

- 必须保持不变的元素。
- 允许动态变化的元素。
- 禁止 AI 背景生成的元素。
- HTML 内容如何叠加到背景之上。

### 7. Section 实现策略

如果进入实现模式：

- 一次只实现一个 Section。
- 优先复用现有组件和样式令牌。
- 不切换技术栈。
- 不改变业务承诺。
- 每个 Section 后做截图和响应式检查。

### 8. 验收

- 桌面端截图。
- 移动端截图。
- 文本不重叠。
- 背景媒体非空且不遮挡内容。
- 动效支持降级。
- 构建和测试通过。
- Git diff 只包含预期文件。

## Skill 所需模板

建议放入 `templates/`：

- `reference-video-analysis.md`
- `current-site-audit.md`
- `design-dna.md`
- `reusable-workflow.md`
- `implementation-plan.md`
- `visual-qa-report.md`

## Skill 所需脚本

建议放入 `scripts/`：

- `extract-keyframes.ps1`：Windows PowerShell 版 ffmpeg 抽帧和索引生成。
- `extract-keyframes.sh`：macOS/Linux 版 ffmpeg 抽帧和索引生成。
- `make-contact-sheet.ps1`：把关键帧生成联系表。
- `visual-checklist.md`：手动或 Playwright 检查项。

脚本只负责辅助处理素材，不应修改业务代码。

## SKILL.md 应写入的内容

`SKILL.md` 应包含：

- 触发场景和不触发场景。
- 审计/设计/实现/验收四种模式。
- 保护业务代码和 Git 状态的规则。
- 参考素材处理步骤。
- 设计 DNA 提取维度。
- 禁止复制参考页面具体品牌、文案、Logo、布局的规则。
- 背景媒体必须排除文字、按钮、导航、Logo、假 UI 的规则。
- 真实 HTML 叠加原则。
- 逐个 Section 实现原则。
- 桌面、移动、性能、动效、可访问性的验收清单。
- 输出文档格式。

## 应放入 references 或 scripts 的内容

放入 `references/`：

- 示例关键帧说明。
- 高质量页面的分析样例。
- 背景媒体正例/反例。
- 不含具体项目敏感信息的示例素材。

放入 `scripts/`：

- 抽帧脚本。
- 联系表生成脚本。
- 截图或视觉检查脚本。

不应放入通用 Skill 的内容：

- 当前视频解析网站的访问码。
- 当前服务器地址。
- 当前 GitHub 仓库地址。
- 当前项目的具体平台支持承诺。
- 任何用户 Cookie、密钥或生产环境变量。

## 本项目后续使用建议

后续如果创建 `premium-web-dna-workflow` Skill，本项目调用时应额外提供项目规则：

- 服务器不代下载、不保存、不代理视频。
- 访问码只解锁本站功能，不代表登录视频平台。
- 平台 Cookie 仅用于单次解析。
- 当前真实支持平台以代码为准。
- 未来新增抖音、小红书、微博时，未完成前必须显示待接入。

这样 Skill 可以复用设计方法，而不会把“超级视频解析网站”的业务规则污染到其他项目。
