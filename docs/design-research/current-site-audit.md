# 当前网站审计

## 审计范围

读取范围：

- `package.json`
- `src/app/page.tsx`
- `src/app/layout.tsx`
- `src/app/globals.css`
- `src/components/*`
- `src/lib/*`
- `src/app/api/*`
- `tests/*`
- `references/current-version.png`

本阶段未修改 `src/`、后端解析逻辑或现有页面业务代码。

## 当前技术栈和结构

- 框架：Next.js 15，React 19，TypeScript
- 页面形态：单页控制台
- API：Next Route Handlers，`/api/login`、`/api/logout`、`/api/resolve`
- 解析核心：`yt-dlp` 输出转换为统一 `Manifest`
- 浏览器端能力：`@ffmpeg/ffmpeg` 用于本机浏览器合并
- 样式系统：单个 `globals.css`，使用 CSS 变量、全局类名和响应式媒体查询
- 粒子背景：`ParticleField.tsx` 使用 Canvas，根据窗口和鼠标位置绘制粒子和连线

## 当前信息架构

当前首页由三层组成：

1. 顶部 Hero 说明站点定位：超级视频解析、服务器不代下载、浏览器本机合并、临时 Cookie 不保存。
2. 主控制区两列布局：左侧站内授权，右侧解析控制台。
3. 解析成功后，在右侧解析区下方追加结果视图和下载面板。

当前没有左侧平台模块导航。代码层实际支持的平台是 `x`、`bilibili`、`youtube`，还没有抖音、小红书、微博解析器。

## 核心用户操作路径

公开链接路径：

1. 用户粘贴视频链接。
2. 点击“开始解析”。
3. 服务端调用对应解析器返回媒体清单。
4. 用户选择清晰度或轨道组合。
5. 用户直连源站下载、在浏览器本机合并，或复制本地工具命令。

账号态路径：

1. 用户输入本站访问码。
2. 站点设置 `svr_session` 会话 Cookie。
3. 页面显示平台 Cookie 输入框。
4. 用户粘贴 X/YouTube/Bilibili 等平台 Cookie。
5. Cookie 仅用于本次解析请求，不持久化保存。

## 值得保留的部分

- “服务器不代下载、不保存、不代理”的安全边界表达清楚，适合项目定位。
- 访问码和平台 Cookie 的解释已经比第一版更明确。
- 统一 Manifest 模型有利于后续增加平台。
- 直链、浏览器合并、本地命令三种下载方式划分合理。
- Canvas 粒子背景已经有 `prefers-reduced-motion` 相关配置，基础动效风险可控。
- CSS 变量已包含背景、文字、线框、强调色、危险和警告色，可继续扩展为正式设计令牌。

## 问题清单

### 高优先级

1. 信息架构与“超级视频解析平台”目标不完全匹配

当前页面仍是单页表单工具，没有平台级工作区。用户想要的抖音、Twitter、小红书、微博等模块化体验，需要左侧平台导航、平台状态、平台专属说明、平台专属 Cookie/错误提示和能力差异展示。

2. 主操作被 Hero 文案稀释

Hero 占据大量首屏空间，真正的输入框在右下方。工具型网站的核心价值是快速粘贴和解析，当前首屏更像“产品介绍 + 控制台”，操作效率还可以提高。

3. 视觉层级偏平均

Hero、授权卡片、解析面板都使用类似的玻璃拟态、边框和阴影。所有区域都在发光和加边框，导致视觉重点不够明确。

4. 访问码、平台 Cookie、平台账号态仍需要更强的交互区分

虽然文案已说明访问码不是平台登录，但 UI 上仍然把“站内授权”和“平台 Cookie”放在相近的卡片语言里。后续加入更多平台后，这个概念如果不模块化，用户会继续困惑。

5. 平台支持状态没有前置

当前能力条写着 YouTube/Bilibili/X，但用户无法在输入前选择平台，也看不到平台可用性、Cookie 要求、常见失败原因。对于 Twitter/X 一类容易失败的平台，缺少预防性说明。

### 中优先级

1. 字体系统偏基础

全站使用 Arial/Helvetica/sans-serif。中文显示可用，但缺少更适合中文工具界面的字体栈和字重节奏，导致“高级控制台”感主要依赖暗色和发光，而不是排版质量。

2. 色彩接近单一霓虹科技风

当前主色集中在深蓝黑、青绿、少量琥珀和粉色。虽然符合技术感，但如果继续加强粒子和发光，很容易变成模板化的“赛博控制台”。

3. 间距和布局密度不够像高频工具

Hero 和面板之间留白较大，控制区没有形成左侧模块、顶部状态、中央工作区、右侧结果的稳定工作台结构。对频繁解析用户来说，首屏空间利用率偏低。

4. 卡片数量和边框权重偏多

当前页面的 Hero、授权、解析、结果、下载都以卡片呈现。卡片是有效容器，但如果未来再加平台模块、历史记录、帮助说明，很容易出现“卡片套卡片”的模板感。

5. 输入区域缺少平台感知

输入框 placeholder 只列 X 和 Bilibili。对于 YouTube、抖音、小红书、微博等未来平台，应由选中平台决定提示、Cookie 示例、风险说明和错误恢复建议。

6. 错误结果缺少操作恢复路径

错误只显示文本。例如 X/Twitter 未找到视频时，下一步应该提示检查是否引用帖、是否需要 Cookie、是否复制了帖子详情页、是否为图片帖或敏感内容。

### 低优先级

1. 部分 UI 缺少图标辅助

按钮和能力标签都是纯文本。对于下载、合并、复制、平台选择等工具操作，图标可以降低扫描成本。

2. `ResultView` 有少量 inline style

缩略图和变体按钮有内联样式，后续设计系统化时应收敛到 CSS 类或组件样式。

3. 全局 CSS 文件会继续膨胀

目前规模可控，但若加入平台工作区、模块切换、结果列表、历史记录、状态条，建议拆分组件样式或建立更明确的样式分层。

4. 没有视觉验收脚本

已有 Vitest 覆盖解析和粒子配置，但没有针对桌面/移动截图、Canvas 非空、文本不重叠的视觉检查流程。

## 区域审计

### 首屏

优点：定位清楚，标题醒目，安全边界明确。

问题：主操作没有成为第一视觉焦点；顶部 Hero 更像宣传区。对于工具型 Web 应用，可以把 Hero 收敛成状态头，释放空间给平台工作区和解析结果。

### 输入区域

优点：输入框、按钮、Cookie 文本域结构清楚；按钮状态有 loading。

问题：缺少平台选择、平台专属提示、示例链接、粘贴校验、错误恢复入口。按钮视觉较重，但输入框本身没有足够的“命令中心”气质。

### Cookie 区域

优点：已经明确“这里填的是平台 Cookie，不是本站访问码”。

问题：不同平台 Cookie 获取方式不同，未来不应放在一个通用 textarea 里。更适合按平台展示：需要哪些 Cookie、支持 header 还是 Netscape、是否仅本次请求、常见错误。

### 解析结果区域

优点：Manifest 模型清楚，能展示标题、作者、时长、轨道数、变体和下载方式。

问题：结果是在表单下方追加，不像主工作台。解析成功后应强化“结果工作区”，例如质量排序、推荐下载方式、轨道类型、源站限制、复制命令、安全提醒。

## 与工具型 Web 应用定位的匹配度

当前方向基本匹配：暗色控制台、清晰安全边界、直接表单操作，适合技术工具。

但距离“高级视频解析平台”还差一个信息架构层：平台模块、状态面板、结果工作区、平台专属说明和高频操作效率。当前视觉更像一个漂亮 MVP，而不是可长期使用的多平台解析工作台。

## 问题分类

### 视觉设计问题

- 层级平均，很多区域都使用类似边框、阴影和发光。
- 色彩偏单一科技风，缺少平台识别和功能状态色的系统化使用。
- 字体系统基础，中文排版缺少更细的字重、字号和行高策略。
- 卡片数量未来容易膨胀，页面会产生模板感。
- 背景网格和粒子气质明确，但需要克制，否则会干扰工具效率。

### 交互问题

- 没有平台切换和平台专属流程。
- 访问码、平台 Cookie、账号态的路径还没有完全模块化。
- 错误提示缺少下一步动作。
- 解析结果没有形成独立工作区。
- 未来加入抖音、小红书、微博时，用户需要看到支持状态和限制。

### 代码结构问题

- `PlatformSchema` 只包含 `x`、`bilibili`、`youtube`，平台扩展会牵动 Manifest、检测、解析器和 UI。
- 平台能力条是静态字符串，没有和实际解析器能力绑定。
- 样式集中在 `globals.css`，扩展平台工作区后可维护性会下降。
- 结果组件中存在 inline style，后续统一视觉系统时应收敛。
- 当前没有视觉/响应式自动验收脚本。

## A. 可复用于其他网站的通用方法

- 先审计信息架构和核心路径，再评价视觉。
- 把问题分为视觉、交互、代码结构三类，避免只改颜色和阴影。
- 保护已有正确业务边界，把视觉升级限制在不改变承诺的范围内。
- 判断页面是“展示型”还是“工具型”，再决定 Hero 和工作区比例。
- 建立可验证的视觉问题清单，而不是只说“更高级”。

## B. 仅适用于超级视频解析网站的项目规则

- 服务器不代理、不保存、不下载视频文件是最高优先级承诺。
- 访问码只代表本站授权，不代表登录 X/YouTube/Bilibili 等平台。
- 平台 Cookie 只用于单次解析请求，不应持久化。
- 结果区必须清晰区分源站直链、浏览器本地合并、本地工具命令。
- 未实现的平台不能在解析能力上假装可用。
- X/Twitter 等易失败平台需要更强的前置说明和错误恢复路径。
